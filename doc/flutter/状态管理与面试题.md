# Flutter状态管理与面试题

状态管理是 Flutter 面试中最常见模块，重点不是背库，而是讲清状态边界和方案取舍。

## 1. 什么是状态？

- 会随时间变化并影响 UI 的数据都属于状态。
- 常见分为：页面局部状态、页面共享状态、全局业务状态。

## 2. `setState` 的适用边界

- 适合页面内部、影响范围小、逻辑简单的状态。
- 不适合跨页面共享和复杂业务编排。

## 3. 常见状态管理方案

- Provider：学习成本低，适合中小项目。
- Riverpod：依赖管理清晰，可测试性好。
- Bloc/Cubit：事件流清晰，适合团队协作与大型项目。
- GetX：开发效率高，但需约束规范避免隐式耦合。

## 4. 状态管理选型思路

- 看团队熟悉度和维护成本。
- 看业务复杂度与模块规模。
- 看测试需求与可观测性要求。

## 5. 与生命周期协同

- 页面销毁后避免继续更新 UI。
- 异步回调中先判断 `mounted`。
- 流和订阅要在合适时机释放，避免内存泄漏。

## 6. 高频面试题

### Q1: Provider 和 Riverpod 最大区别是什么？
A: Riverpod 更强调编译期安全和依赖声明，解耦 `BuildContext`，测试体验更好。

### Q2: 为什么不建议“全项目都用 setState”？
A: 复杂业务下状态边界不清、可维护性差，容易出现耦合与重复构建问题。

### Q3: Bloc 的价值是什么？
A: 事件驱动 + 单向数据流让状态变化可追踪，适合复杂交互和多人协作。

### Q4: 如何避免状态管理过度设计？
A: 从简单到复杂演进，先局部状态，再按共享范围逐步抽离。

## 7. 小结

状态管理的面试高分点是“边界清楚、取舍合理、能落地”，而不是单纯对比框架语法。
