# 组合式 API 与面试题

## 知识点

- Composition API 通过 `setup` / `<script setup>` 组织逻辑，按功能聚合而不是按选项分散。
- 核心收益：复用逻辑更清晰、类型推导更友好、复杂页面可维护性更高。
- 常用能力：`ref`、`reactive`、`computed`、`watch`、`onMounted`、自定义 `composable`。

## 实战建议

- 通用逻辑抽到 `useXxx` 中，返回状态与行为。
- 组件尽量只组合 `composable`，避免把所有逻辑堆在一个文件。
- 复杂状态优先 `ref` + 对象拆分，减少深层响应式带来的心智负担。

## 高频面试题

### 1. Composition API 相比 Options API 的本质优势是什么？
答：按“业务能力”组织代码，逻辑复用更自然，类型支持更好，复杂组件更容易维护。

### 2. 为什么很多团队推荐 `<script setup>`？
答：语法更简洁、模板可直接使用变量、编译期优化更充分，样板代码更少。

### 3. 什么时候应该抽 `composable`？
答：当逻辑跨组件复用，或单组件逻辑块过大、可独立测试时。
