# Android 多线程

本文从“原理 + 工程实践 + 面试深度”三个层面系统整理 Android 多线程知识。包含关键原理、常见陷阱、代码案例与配图。

![](/doc/Android/assets/thread-model.svg)

## 1. 线程模型与基本规则

- Android 只有主线程可以更新 UI。
- 主线程同时负责渲染、事件分发与生命周期回调。
- 主线程阻塞超过 5s 可能触发 ANR。

核心结论：所有耗时任务必须放到后台线程，并正确切回主线程更新 UI。

## 2. Java 内存模型（JMM）与 happens-before

这是面试高频重点，理解“可见性、原子性、有序性”。

- 可见性：一个线程修改变量，另一个线程能看到。
- 原子性：操作不可被中断，执行要么成功要么失败。
- 有序性：指令重排不会破坏线程间语义。

happens-before 规则（精简版）：
- 同一线程内，程序顺序规则。
- `synchronized` 解锁 happens-before 后续加锁。
- `volatile` 写 happens-before 后续读。
- 线程启动 `start()` happens-before 线程内任何操作。
- 线程结束 `join()` happens-before 主线程后续操作。

## 3. 线程生命周期与状态

- `NEW`：创建未启动。
- `RUNNABLE`：运行或就绪。
- `BLOCKED`：等待锁。
- `WAITING`：等待唤醒（如 `wait()`）。
- `TIMED_WAITING`：超时等待（如 `sleep()`）。
- `TERMINATED`：结束。

理解线程状态有助于定位卡顿、死锁与资源泄漏。

## 4. 主线程切换方式

常见切回主线程的方式：

```kotlin
Handler(Looper.getMainLooper()).post {
    // 更新 UI
}
```

```kotlin
lifecycleScope.launch(Dispatchers.Main) {
    // 更新 UI
}
```

```kotlin
runOnUiThread {
    // 更新 UI
}
```

## 5. Handler/Looper/MessageQueue 深度

线程通信核心机制：
- `Looper` 维护消息循环。
- `MessageQueue` 负责消息入队。
- `Handler` 发送消息、处理消息。

典型模式：在后台线程执行耗时任务，完成后通过 `Handler` 通知主线程。

**案例：使用静态 Handler + WeakReference 防止泄漏**

```kotlin
class SafeHandler(activity: Activity) : Handler(Looper.getMainLooper()) {
    private val ref = WeakReference(activity)
    override fun handleMessage(msg: Message) {
        ref.get()?.let {
            // 更新 UI
        }
    }
}
```

## 6. 线程池原理与调优

线程池解决两个核心问题：
- 线程复用，避免频繁创建销毁。
- 控制并发，避免过载。

![](/doc/Android/assets/thread-pool.svg)

核心参数：
- `corePoolSize` 核心线程数。
- `maximumPoolSize` 最大线程数。
- `workQueue` 任务队列。
- `RejectedExecutionHandler` 拒绝策略。

常见队列选择：
- `LinkedBlockingQueue`：无界队列，任务堆积风险高。
- `ArrayBlockingQueue`：有界队列，可控。
- `SynchronousQueue`：不缓存任务，直接交给线程。

**面试要点：**
- 线程池的核心任务是“控制并发与保护系统资源”。 
- 线程池大小建议：CPU 密集型使用 `NCPU + 1`，IO 密集型使用 `2 * NCPU` 到 `4 * NCPU`。

**案例：自定义线程池**

```kotlin
val pool = ThreadPoolExecutor(
    2,
    4,
    30,
    TimeUnit.SECONDS,
    ArrayBlockingQueue(100),
    ThreadPoolExecutor.CallerRunsPolicy()
)

pool.execute {
    // 后台任务
}
```

## 7. Kotlin 协程深度

协程本质是“轻量级线程 + 调度器”。

重点概念：
- `Dispatchers.Main` 主线程调度。
- `Dispatchers.IO` 适合 IO。
- `Dispatchers.Default` 适合 CPU 计算。
- 结构化并发：子协程生命周期受父协程管理。

**案例：结构化并发 + 异常处理**

```kotlin
viewModelScope.launch {
    supervisorScope {
        val a = async(Dispatchers.IO) { fetchA() }
        val b = async(Dispatchers.IO) { fetchB() }
        try {
            val result = a.await() + b.await()
        } catch (e: Exception) {
            // 处理异常
        }
    }
}
```

## 8. RxJava 线程切换

- `subscribeOn` 控制上游线程。
- `observeOn` 控制下游线程。
- 订阅释放 `dispose()`。

面试高频：`subscribeOn` 只生效一次，离源头最近的有效。

## 9. WorkManager 与后台任务

当任务需要跨进程、跨重启时，用 `WorkManager`：

- 适合可靠任务调度。
- 支持网络、充电等约束条件。

## 10. 并发问题与解决方案

### 10.1 竞态条件

```kotlin
var count = 0
fun inc() {
    count++
}
```

解决方案：
- 用 `synchronized`。
- 用 `AtomicInteger`。

### 10.2 死锁

典型原因：多个锁交叉嵌套且获取顺序不一致。

解决方案：
- 保持一致的加锁顺序。
- 尽量减少锁数量。

### 10.3 可见性问题

```kotlin
@Volatile private var running = true
```

- `volatile` 解决可见性问题，但不保证原子性。

## 11. ANR 排查与实践

- 主线程卡顿超过 5s 易触发 ANR。
- 检查 `traces.txt` 线程堆栈。
- 避免在 `onCreate()` 执行耗时操作。
- 使用 `StrictMode` 发现主线程 IO。

**案例：StrictMode 配置**

```kotlin
if (BuildConfig.DEBUG) {
    StrictMode.setThreadPolicy(
        StrictMode.ThreadPolicy.Builder()
            .detectAll()
            .penaltyLog()
            .build()
    )
}
```

## 12. 常见面试问题与答法

### Q1: 为什么主线程不能执行耗时任务？
- 因为主线程负责事件分发、页面绘制和生命周期回调，一旦被耗时任务阻塞，输入事件无法及时处理，渲染帧也无法按时提交。
- 结果就是卡顿，严重时触发 ANR。面试里最好补充一句：本质上是消息队列被长任务占住了。

### Q2: 线程池核心参数有哪些？怎么调优？
- 重点参数是 `corePoolSize`、`maximumPoolSize`、`keepAliveTime`、阻塞队列和拒绝策略。
- CPU 密集型任务通常线程数接近 CPU 核数，IO 密集型任务可以适当放大，但前提是有界队列和明确降级策略，避免任务无限堆积。

### Q3: `volatile` 和 `synchronized` 的区别是什么？
- `volatile` 解决的是可见性和有序性问题，不能保证复合操作的原子性。
- `synchronized` 既能保证可见性，也能保证同一时刻只有一个线程进入临界区，因此还能解决原子性问题，但开销相对更高。

### Q4: `Handler` 为什么容易引发内存泄漏？如何避免？
- 非静态内部类 `Handler` 会默认持有外部 Activity/Fragment 引用，如果消息在队列里停留太久，而页面已经销毁，就会导致页面对象无法回收。
- 常见做法是使用静态内部类 + `WeakReference`，或者直接使用与生命周期绑定的协程、`lifecycleScope` 替代。

### Q5: 协程的结构化并发是什么？
- 结构化并发强调“子任务不能脱离父任务独立失控”，子协程生命周期受父协程管理。
- 这带来的好处是取消传播明确、异常处理集中、资源更容易回收，所以 Android 中通常在 `viewModelScope` 和 `lifecycleScope` 内发起协程。

### Q6: `Dispatchers.IO` 和 `Dispatchers.Default` 怎么区分？
- `Default` 适合 CPU 密集型任务，比如排序、解析、计算。
- `IO` 适合数据库、文件、网络请求这类阻塞式 IO，它允许更高的并发上限，避免少量阻塞任务把计算线程全部占满。

### Q7: ANR 一般是怎么产生的？如何排查？
- 常见原因是主线程长时间执行 IO、锁等待、Binder 调用阻塞，或者 `BroadcastReceiver`、Service、输入事件处理超时。
- 排查时先看 `traces.txt` 主线程栈，再结合 `StrictMode`、Profiler、Perfetto 判断是 CPU 忙、锁竞争还是主线程误用。

## 13. 最佳实践总结

- 优先使用协程 + `lifecycleScope`/`viewModelScope`。
- 线程池使用有界队列，避免无限堆积。
- 明确任务生命周期，避免泄漏。
- 大任务拆分，避免长期占用主线程。

## 14. 小结

深刻理解多线程，需要从“JMM 原理 + 线程通信 + 调度策略 + 生命周期管理”四个维度入手。面试中重点考察的是：线程池调优、协程结构化并发、可见性/原子性问题以及 ANR 排查思路。
