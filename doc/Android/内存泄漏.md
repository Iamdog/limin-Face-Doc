# Android 内存泄漏

本文汇总 Android 开发中常见的内存泄漏场景，给出原因与解决方案，便于排查与规避。

## 1. 静态持有 Activity 或 View

原因：`static` 字段持有 `Activity`、`View`、`Context`，导致对象无法被回收。

解决方案：
- 避免在静态变量中保存 `Activity` 或 `View`。
- 需要全局对象时，使用 `Application` 级别 `Context`。
- 确实需要缓存 View 时，用 `WeakReference` 并主动清理。

## 2. 匿名内部类/非静态内部类持有外部引用

原因：`Handler`、`Runnable`、`Thread` 等匿名内部类默认持有外部类（Activity/Fragment）引用，任务执行时间过长导致泄漏。

解决方案：
- 将内部类改为 `static`，并通过 `WeakReference` 引用外部对象。
- 在 `onDestroy()` 取消消息与任务。
- 使用 `Lifecycle` 感知组件或 `CoroutineScope`/`viewModelScope` 管理生命周期。

## 3. Handler 消息未移除

原因：`Handler` 消息队列中仍有 `Message`/`Runnable`，引用了外部对象。

解决方案：
- `onDestroy()` 中调用 `handler.removeCallbacksAndMessages(null)`。
- 使用静态 `Handler` + `WeakReference`。
- 优先使用 `Lifecycle` 感知的事件流替代手写 `Handler`。

## 4. 线程/异步任务生命周期过长

原因：`Thread`、`Timer`、`AsyncTask`、`Executor` 持有 `Activity`，任务未停止或未取消。

解决方案：
- `onDestroy()` 中停止线程或取消任务。
- 使用 `CoroutineScope` 与 `lifecycleScope`。
- 避免在后台任务中强引用 `Activity`。

## 5. 监听器/回调未解绑

原因：注册到系统服务或第三方 SDK 的监听器持有 `Activity` 引用，未在销毁时注销。

解决方案：
- 在 `onPause()`/`onStop()`/`onDestroy()` 中明确 `unregister`/`removeListener`。
- 对需要长期监听的对象使用 `Application` Context。

## 6. 广播接收器未注销

原因：动态注册的 `BroadcastReceiver` 未在生命周期内注销。

解决方案：
- 在 `onDestroy()` 中调用 `unregisterReceiver()`。
- 优先使用 `LocalBroadcastManager` 或 `LiveData`/`Flow`。

## 7. ContentObserver/NotificationListener 未释放

原因：注册了系统观察者，但未注销，导致引用链保留。

解决方案：
- 在 `onDestroy()` 或 `onStop()` 中调用 `unregisterContentObserver()`。
- 避免在短生命周期组件中注册全局观察者。

## 8. Bitmap/Drawable 未正确回收或复用

原因：大图加载导致 `Bitmap` 持有大量内存，或 `ImageView` 的 `Drawable` 未及时释放。

解决方案：
- 使用图片加载库（Glide/Coil/Picasso）并启用缓存与生命周期管理。
- 及时清空 `ImageView`（如 `imageView.setImageDrawable(null)`）。
- 尽量使用 `inSampleSize` 压缩图片。

## 9. 单例持有短生命周期对象

原因：单例模式对象持有 `Activity`、`View` 或 `Fragment`。

解决方案：
- 单例仅持有 `Application` 级别对象。
- 使用弱引用或在生命周期结束时主动释放。

## 10. WebView 泄漏

原因：`WebView` 持有 `Activity` 上下文，且内部资源未释放。

解决方案：
- 使用 `Application` Context 创建 `WebView` 或在独立进程中使用。
- `onDestroy()` 中调用 `webView.destroy()` 并移除父布局引用。
- 清空 `WebView` 数据与历史。

## 11. Cursor/IO 未关闭

原因：`Cursor`、`InputStream`、`OutputStream` 等未关闭导致资源占用。

解决方案：
- 使用 `try-with-resources` 自动关闭。
- 确保异常路径同样关闭资源。

## 12. 集合缓存无限增长

原因：静态 `List`/`Map` 持续累积对象，缺少淘汰策略。

解决方案：
- 使用 `LruCache` 或限制集合容量。
- 定期清理缓存或基于生命周期释放。

## 13. Adapter/ViewHolder 持有 Context

原因：`Adapter` 中持有 `Activity` 引用且生命周期长于页面。

解决方案：
- 使用 `Context` 的弱引用。
- `Adapter` 不再使用时置空引用。

## 14. RxJava 订阅未释放

原因：`Disposable` 未 `dispose()`，导致链路持有 `Activity`。

解决方案：
- `onDestroy()` 中统一 `dispose()`。
- 使用 `AutoDispose` 或 `lifecycle` 绑定。

## 15. 自定义 View 中启动线程/动画

原因：自定义 View 内部启动线程/动画，View 销毁后仍在运行。

解决方案：
- 在 `onDetachedFromWindow()` 中停止线程与动画。
- 使用 `LifecycleObserver` 管理相关资源。

## 16. Fragment View 泄漏

原因：Fragment 的 `onDestroyView()` 后仍持有 `View` 引用。

解决方案：
- 在 `onDestroyView()` 将 `View`、`Adapter` 等引用置空。
- 使用 `viewLifecycleOwner` 绑定观察者。

## 17. 静态资源持有 Context 引用链

原因：工具类、缓存类中静态字段持有 `Context` 或 `View`。

解决方案：
- 改为传参使用或使用 `Application` Context。
- 使用 `WeakReference` 并在适当时机释放。

## 18. 第三方 SDK 引用未释放

原因：SDK 监听器/回调注册后未注销，或内部持有 `Activity`。

解决方案：
- 查看 SDK 文档并在生命周期内注销。
- 使用 `Application` Context 进行初始化。

## 19. 动画与 Animator 泄漏

原因：`ObjectAnimator`/`ValueAnimator` 未停止，持有 View。

解决方案：
- 在 `onPause()`/`onDestroy()` 中 `cancel()`。
- 避免在后台保持长时间动画。

## 20. 资源引用与缓存冲突

原因：资源缓存与页面生命周期不一致，导致引用链保留。

解决方案：
- 清晰区分全局缓存与页面缓存。
- 页面级缓存随生命周期释放。

## 排查工具与建议

- 使用 Android Studio Memory Profiler 定位引用链。
- 引入 LeakCanary 自动检测内存泄漏。
- 在关键生命周期打印日志，确认注册/解绑匹配。
