# Android 线程池调优实战

线程池题考察的是“并发控制能力”，不是背参数定义。要能结合业务场景给出可落地配置。

## 1. 线程池核心参数

- `corePoolSize`
- `maximumPoolSize`
- `keepAliveTime`
- `workQueue`
- `RejectedExecutionHandler`

## 2. 调优原则

- CPU 密集型：线程数接近 CPU 核数。
- IO 密集型：线程数可适当放大，但必须有队列上限。
- 高峰期要有拒绝与降级策略。

## 3. 队列选择

- `ArrayBlockingQueue`：有界，强约束。
- `LinkedBlockingQueue`：默认近似无界，易堆积。
- `SynchronousQueue`：不缓存任务，直接交付执行线程。

## 4. 常见事故模式

- 任务堆积导致 OOM。
- 线程数过大导致上下文切换开销暴涨。
- 主线程等待线程池任务结果造成 ANR。

## 5. 实战建议

- 按业务拆分线程池，避免所有任务混用。
- 打点监控队列长度、活跃线程数、任务耗时分位值。
- 关键路径任务设置超时和降级。

## 6. 常见面试题

### Q1: 为什么不建议默认用 `Executors.newCachedThreadPool()`？
A: 因为线程增长不受控，峰值时容易把系统资源打满。

### Q2: 拒绝策略怎么选？
A: 看业务容忍度。可丢任务选丢弃策略，不可丢任务可考虑调用方执行或入持久队列。

### Q3: 如何判断线程池是否健康？
A: 看任务排队时长、拒绝率、线程利用率和高峰期尾延迟，而不是只看平均耗时。

### Q4: 单线程池什么时候有价值？
A: 任务必须串行且顺序敏感时，比如日志落盘、状态机驱动任务。

### Q5: 线程池调优第一步是什么？
A: 先量化任务类型和负载，再做配置，不要拍脑袋定参数。

## 7. 小结

线程池调优本质是“用有限资源保证关键任务完成”，重点是上限、监控和降级，不是盲目提并发。
