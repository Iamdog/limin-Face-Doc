# Android OOM专题

OOM 是 Android 稳定性核心问题。面试重点不是定义，而是“为什么发生、如何定位、如何治理”。

## 1. 什么是 OOM？

- OOM 是 `OutOfMemoryError`，表示进程可用内存不足，无法再分配对象。
- 常见于大图、缓存失控、对象泄漏、Native 内存增长。

## 2. 高频触发场景

- 一次性加载大图或多图。
- 列表图片缓存策略失控。
- 页面频繁创建大对象导致内存峰值抬升。
- Activity/Fragment 泄漏导致旧页面堆积。
- JNI/Bitmap 等 Native 内存未释放。

## 3. 排查路径

1. 先看崩溃堆栈，确认是 Java 堆还是 Native 堆问题。
2. 用 Memory Profiler 看峰值增长曲线。
3. 用 Heap Dump 查主导对象和引用链。
4. 对照业务场景复现：列表滑动、相册选择、长时间后台等。

## 4. 治理策略

- 图片按需解码，控制尺寸与格式。
- 缓存设置上限，避免无限增长。
- 大对象分批加载，避免峰值集中。
- 生命周期结束及时释放引用。
- 引入 LeakCanary 做持续检测。

## 5. `largeHeap` 要不要用？

- 可以作为兜底，不应当作根治方案。
- `largeHeap` 会掩盖问题并增加系统整体内存压力。

## 6. 面试高频问题

### Q1: OOM 和内存泄漏有什么关系？
A: 泄漏不一定立刻 OOM，但会持续抬高基线，最终显著提高 OOM 概率。

### Q2: 为什么图片最容易引发 OOM？
A: 因为图片内存占用按像素展开，常远大于文件体积。

### Q3: Java 堆没满也会 OOM 吗？
A: 会，尤其是 Native 内存、线程栈或大对象连续分配失败时。

### Q4: 如何证明是缓存导致 OOM？
A: 看堆快照中的缓存容器对象数量和 retained size 是否长期上升且不回落。

### Q5: OOM 治理的第一优先级是什么？
A: 先定位主导来源，再做限流和上限控制，避免“盲目清理一切缓存”。

## 7. 小结

OOM 治理要以“峰值控制 + 泄漏治理 + 场景复现”三板斧推进，先稳住崩溃率，再逐步优化体验和性能。
