# Android 性能优化

本文聚焦 Android 开发中**性能优化问题与解决方案**，同时给出**面试高频问题与答法**。内容覆盖大厂与小厂常见场景，强调可落地的诊断路径与权衡策略。

## 1. 性能优化的核心思路（可复用流程）

1. **先定位再优化**：没有数据支撑的优化都是猜测。
2. **明确指标**：启动时长、帧率（Jank）、内存峰值、ANR/Crash、耗电、流量等。
3. **分层分析**：应用层 → 框架层 → 系统层 → 设备层。
4. **只做 ROI 高的优化**：让“最痛的用户场景”先变好。

## 2. 常见性能瓶颈分类

- **启动性能**：冷启动/温启动/热启动耗时过长。
- **UI 卡顿**：掉帧、长帧、布局过深、渲染过度。
- **内存问题**：内存抖动、频繁 GC、泄漏、OOM。
- **IO 问题**：磁盘 IO、数据库慢查询、主线程阻塞。
- **网络问题**：接口慢、首包慢、弱网重试风暴。
- **线程与并发**：线程过多、锁竞争、任务调度失衡。
- **耗电问题**：后台唤醒频繁、定位/传感器滥用。

## 3. 诊断与量化工具

- **Android Studio Profiler**：CPU/Memory/Network/Frame。
- **Systrace/Perfetto**：系统级卡顿、线程调度、IO。
- **FrameMetrics / JankStats**：定位掉帧和长帧。
- **StrictMode**：主线程 IO/网络监测。
- **LeakCanary**：内存泄漏检测。
- **Firebase Performance / 自研埋点**：线上真实耗时。

## 4. 常见问题与解决方案（详细）

### 4.1 启动慢（冷启动为主）

**原因**：
- `Application` 做了太多初始化。
- 首屏布局复杂，Image/字体/资源加载重。
- 首屏依赖网络请求，接口慢导致阻塞。

**解决方案**：
- **分阶段初始化**：主线程只做必需项（如 Crash/日志），其他延后到首帧之后。
- **首屏数据降级**：默认占位 + 异步刷新。
- **延迟初始化与按需加载**：用 `App Startup` 或自定义任务调度。
- **瘦身资源**：压缩图片、避免大字体/大图首屏。

**独到见解**：
- 大厂普遍用“启动任务编排”，明确优先级与依赖；小厂至少要做到**首屏解耦**与**懒加载**。

### 4.2 页面卡顿 / 掉帧

**原因**：
- 布局层级过深、复杂约束（Constraint 过度）。
- `onDraw` / `onLayout` 做了大量计算。
- RecyclerView item 过重或频繁 `notifyDataSetChanged`。

**解决方案**：
- **布局扁平化**：减少嵌套，避免不必要的层级。
- **避免每帧计算**：缓存测量结果，减少 `onDraw` 逻辑。
- **列表优化**：稳定 ID、DiffUtil、避免频繁重绘。
- **图片渲染优化**：合适尺寸+占位，避免过度解码。

**独到见解**：
- 真正的卡顿通常是“多个小问题叠加”，不要只盯某一个耗时函数。

### 4.3 ANR（主线程阻塞）

**原因**：
- 主线程执行网络/IO。
- 过长 `onCreate` / `onResume`。
- 广播、Service 执行过重。

**解决方案**：
- 严禁主线程 IO，使用线程池/协程。
- 生命周期中只保留轻逻辑。
- 使用 `StrictMode` 发现阻塞点。

### 4.4 内存抖动与频繁 GC

**原因**：
- 频繁创建短生命周期对象。
- Bitmap 重复创建。

**解决方案**：
- 对象复用（池化）、避免临时对象。
- 图片缓存与解码优化（比如采样率）。

### 4.5 OOM 与内存峰值过高

**原因**：
- 大图/大文件一次性加载。
- 多进程共享数据不当。

**解决方案**：
- 分段加载/流式处理。
- 合理配置 `largeHeap`，但不依赖它。
- 监控内存峰值并限制缓存上限。

### 4.6 网络慢与弱网体验差

**原因**：
- 首包慢，接口未优化。
- 弱网重试策略不合理导致放大请求。

**解决方案**：
- 开启 HTTP/2/3、压缩、缓存。
- 重试策略要**指数退避**。
- 请求合并、分页、预取。

### 4.7 数据库与磁盘 IO 慢

**原因**：
- 未建索引，慢查询。
- 主线程操作数据库。

**解决方案**：
- 使用索引与分页查询。
- 数据库操作统一异步化。
- 监控慢查询日志。

### 4.8 耗电问题

**原因**：
- 后台任务频繁拉活。
- 位置/传感器一直运行。

**解决方案**：
- 使用 `WorkManager` 合理调度。
- 限制后台频率，避免高频唤醒。

### 4.9 线程与并发问题

**原因**：
- 线程池过大或过小。
- 锁竞争导致线程阻塞。

**解决方案**：
- 统一线程池，限制并发。
- 避免粗粒度锁，拆分锁范围。

## 5. 面试高频问题与答法

### Q1: 如何优化冷启动？
- 精简 `Application` 初始化，只保留核心功能。
- 关键依赖放首帧之后初始化。
- 首屏布局轻量化，异步加载非关键资源。

### Q2: 如何排查 UI 卡顿？
- `Profiler` / `JankStats` 定位长帧。
- 查看 `Choreographer`/`FrameMetrics`。
- 优化布局与 `RecyclerView` item。

### Q3: 如何避免 ANR？
- 主线程禁止网络和 IO。
- 生命周期回调中只做轻逻辑。
- 统一线程调度和超时控制。

### Q4: 内存抖动如何解决？
- 避免频繁创建临时对象。
- 引入对象池和缓存。

### Q5: 为什么说优化需要“场景优先”？
- 因为性能优化的最终目标是**用户体验**，不是指标漂亮。
- 先修复“用户高频场景”的痛点，ROI 最大。

## 6. 大厂与小厂优化策略差异

- **大厂**：体系化工具链 + 线上监控 + 灰度验证，优化更多是“收益可控”。
- **小厂**：更看重“首屏与核心流程”，需要**小成本快速提升体验**。

> 总结：大厂做“系统工程”，小厂做“重点突破”，但底层方法论一致。

## 7. 推荐实践清单（可落地）

- 首屏启动拆分 + 延迟初始化。
- 严格禁止主线程 IO。
- 列表布局扁平化 + DiffUtil。
- 图片按需加载 + 合理缓存。
- 线上埋点监控启动、卡顿、ANR、内存峰值。
