## View的生命周期与绘制流程



### 绘制流程

1. **Measure（测量）**：确定 View 的大小（宽和高）。涉及 `MeasureSpec`（由父布局规格和自身 LayoutParams 决定）。**MeasureSpec三种模式**  `EXACTLY`（精确值）、`AT_MOST`（最大值/wrap_content）、`UNSPECIFIED`（不确定）。
2. **Layout（布局）**：确定 View 在父容器中的位置（四个点的坐标），`onLayout` 方法在 `ViewGroup` 中是抽象的，必须由子类实现（如 `LinearLayout` 决定子 View 是横着排还是竖着排）。
3. **Draw（绘制）**：将 View 画到屏幕上。涉及 `Canvas`（画布）、`Paint`（画笔）。① 绘制背景 -> ② 绘制自己（`onDraw`）-> ③ 绘制子 View（`dispatchDraw`）-> ④ 绘制装饰（如滚动条）。



## 触摸事件分发机制

事件分发主要涉及三个方法：`dispatchTouchEvent` (分发)、`onInterceptTouchEvent` (拦截) 和 `onTouchEvent` (消费)。



- **传递链条**：Activity->PhoneWindow->DecorView（作为FrameLayout子类）->ViewGroup->View。



- **分发逻辑（U型走势）**：

  - 向下分发：父容器调用 `onInterceptTouchEvent`。如果返回 `true`，事件被拦截，交给自己的 `onTouchEvent`；如果返回 `false`，则继续传给子 View。
  - 向上回传 ：如果最底层的子View在 `onTouchEvent` 中返回 `false`（不消费），事件会逐级向上传回父容器的 `onTouchEvent`。

  

- **滑动冲突解决**：

  - 外部拦截法：父容器在 `onInterceptTouchEvent` 中判断滑动方向，如果是自己需要的（如左右滑动），就拦截；否则不拦截。
  - **内部拦截法**：子 View 通过 `requestDisallowInterceptTouchEvent(true)` 强行阻止父容器拦截。



## 自定义View

**处理 `wrap_content` 和 `padding`**：

- 直接继承自 View 的控件，必须在 `onMeasure` 中处理 `AT_MOST` 模式，否则 `wrap_content` 会失效（表现为 `match_parent`）。
- 在 `onDraw` 中手动计算内边距（Padding），否则布局设置的 Padding 无效。

**避免在 `onDraw` 中创建对象**：

- `onDraw` 可能会被频繁调用（每秒 60/120 次）。严禁在此处 `new Paint()` 或 `new Path()`，应在构造函数中初始化，避免频繁 GC 导致掉帧。

**减少过度绘制**：

- 使用 `canvas.clipRect()` 剪裁不需要绘制的区域。
- 如果 View 是完全不透明且铺满的，可以移除 XML 中的 `android:background`。

**状态保存**：

- 重写 `onSaveInstanceState` 和 `onRestoreInstanceState`，防止 Activity 重建（如旋屏）导致 UI 状态丢失。



**问：`getWidth()` 和 `getMeasuredWidth()` 有什么区别？**

- **答**：`getMeasuredWidth` 在 `measure` 阶段后就有值；`getWidth` 在 `layout` 阶段后才有值。绝大多数情况下相等，除非在 `layout` 中手动传入了不同的坐标。

**问：`invalidate()` 和 `postInvalidate()` 的区别？**

- **答**：`invalidate()` 必须在 UI 线程调用；`postInvalidate()` 可以在子线程调用。



### 1. 赋值时机（核心区别）

- **`getMeasuredWidth()`**：在 **Measure（测量）**阶段完成后就有值了。具体来说，是在 `setMeasuredDimension()` 方法调用后确定的。
- **`getWidth()`**：在 **Layout（布局）**阶段完成后才有值。它是根据 `layout(l, t, r, b)` 传入的左右坐标计算得出的（width = right - left）。

------

### 2. 定义与意义

- **`getMeasuredWidth()` (期望宽度)**：
  - 表示 View **想要**多大。
  - 它是父 View 根据 `MeasureSpec` 约束和子 View 自身的 `LayoutParams` 测量出来的结果。
- **`getWidth()` (实际宽度)**：
  - 表示 View 在屏幕上**最终显示**多大。
  - 它是父 View 最终给子 View 分配的实际物理尺寸。

------

### 3. 数值不相等的特殊场景

**面试官通常会追问：“什么情况下这两者不相等？”**

虽然 99% 的情况下它们相等，但在以下场景会产生差异：

- **重写 `layout()` 方法**：如果在自定义 `ViewGroup` 中，你在调用子 View 的 `layout(l, t, r, b)` 时，故意传入的 `r - l` 不等于子 View 的 `getMeasuredWidth()`，那么 `getWidth()` 就会显示你手动传入的值。
- **动画/缩放**：注意，`getWidth()` 反映的是布局坐标系的宽度。如果你使用了 `setScaleX()` 进行缩放，`getWidth()` 的值**不会改变**，改变的是 View 的视觉呈现（`getMatrix()` 变换后的结果）。

------

### 总结对比表

| **特性**         | **getMeasuredWidth()**            | **getWidth()**            |
| ---------------- | --------------------------------- | ------------------------- |
| **赋值阶段**     | `onMeasure` 之后                  | `onLayout` 之后           |
| **获取方式**     | 直接读取私有变量 `mMeasuredWidth` | 计算 `mRight - mLeft`     |
| **代表意义**     | View 测量得出的“期望”尺寸         | View 最终占据的“实际”尺寸 |
| **典型调用位置** | 自定义 ViewGroup 的 `onLayout` 中 | `onDraw` 或点击事件处理中 |



## Compose UI

在 Android 开发的演进中，**Jetpack Compose**（声明式）与 **XML**（命令式）代表了两种完全不同的设计哲学。到 2026 年，Compose 已成为主流，但 XML 仍因其稳定性在老旧项目中占有一席之地。

`本人编码过Flutter，该写法与Compose如出一辙`



### 1. 核心作用：将函数转变为 UI 组件

普通的 Kotlin 函数只是执行逻辑，而标记了 `@Composable` 的函数告诉编译器：**“这个函数是用来描述 UI 结构的。”**

- **XML 时代**：UI 是由一个个 `View` 对象构成的树。
- **Compose 时代**：UI 是由一系列 `@Composable` **函数调用**构成的。

------

### 2. 触发“重组”（Recomposition）

这是 `@Composable` 最神奇的地方。当函数内部依赖的状态（State）发生变化时，Compose 框架会自动**重新执行**该函数。

- 它不是刷新整个屏幕，而是智能地对比（Diff）前后的函数输出，只更新变化的那一小块 UI。
- **底层原理**：编译器会修改函数的签名，插入一个 `Composer` 参数，用于记录和跟踪 UI 树的状态。

------

### 3. 类型安全与声明式

- **声明式**：你只需描述“UI 在这个状态下应该是什么样”，而不是“如何去修改 UI”。
- **强类型**：在 XML 中，你传错一个资源 ID 可能只会报运行时错误；但在 `@Composable` 中，参数不匹配在**编译期**就会报错。

------

### 4. 关键特性（面试常考）

如果你在面试中提到以下几点，面试官会觉得你很有深度：

- **无返回值（Unit）**：大部分 Composable 函数不返回值，它们的作用是“产生（Emit）” UI，而不是返回 UI 对象。
- **幂等性（Idempotent）**：同一个 Composable 函数在输入相同参数时，输出的 UI 必须是一致的。
- **无副作用（Side-effect Free）**：你不应该在 Composable 函数内部直接修改全局变量或进行网络请求（除非使用特定的 `LaunchedEffect` 等 API），因为它会被频繁且随机地多次执行。
- **并行执行**：Compose 编译器可能会在多个线程中并行运行不同的 Composable 函数以优化性能，所以不要在函数里写线程不安全的代码。



### 最小实践

```kotlin
import androidx.compose.runtime.*
import androidx.compose.material3.*
import androidx.compose.foundation.layout.*
import androidx.compose.ui.Modifier
import androidx.compose.ui.Alignment

//该注解表示该函数作为了声明式UI组件
@Composable
fun CounterApp() {
    // 1. 声明状态：使用 remember 保证重组时数据不丢失
    var count by remember { mutableIntStateOf(0) }

    // 2. 布局容器：类似 XML 的 RelativeLayout/FrameLayout 居中效果
    Box(
        modifier = Modifier.fillMaxSize(),
        contentAlignment = Alignment.Center
    ) {
        // 3. UI 组件：直接调用函数并传入状态
        Button(onClick = { count++ }) {
            Text(text = "当前计数: $count")
        }
    }
}

```

### 与 XML 的本质区别

| **特性**     | **XML (View System)**        | **@Composable**            |
| ------------ | ---------------------------- | -------------------------- |
| **本质**     | 内存中的 Java 对象（重量级） | 临时的函数执行（轻量级）   |
| **更新方式** | 手动 `setText/setVisibility` | 状态改变自动触发函数重执行 |
| **层级结构** | 静态树状结构                 | 动态代码执行流             |

###  

### XML 与 Compose 的最小对比

如果你用 **XML** 实现同样的功能，你需要：

1. 编写 `layout.xml` (定义 Button 和 TextView)。
2. 在 `Activity` 中 `findViewById`。
3. 定义一个全局变量 `count`。
4. 设置 `setOnClickListener`，并在里面手动调用 `textView.setText()`。

**Compose 的优势**：你只需关注**状态**（count），UI 会自动跟随状态“流动”。这就是所谓的**数据驱动 UI**。



## 布局优化

1. **减少布局层级**：即嵌套层级越多，UI绘制越多越复杂。
2. **减少不必要的背景色设置，如父布局与子布局背景颜色一致，无需两个都设置**
3. **使用include减少重复代码冗余**
4. **merge+include减少include的父层级**
5. **ViewStub进行按需加载，只可以inflate一次**
6. **可使用Layout Inspector，实时查看真机的布局树，寻找嵌套过深的节点**
