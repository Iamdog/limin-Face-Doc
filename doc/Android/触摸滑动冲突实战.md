# Android 触摸滑动冲突实战

这篇只讲实战，不重复事件分发基础。目标是回答“冲突怎么判定、怎么落地、怎么排查”。

## 1. 常见冲突场景

- 外层 `ViewPager2` + 内层横向 `RecyclerView`。
- 外层纵向 `RecyclerView` + 内层可纵向滑动控件。
- `NestedScrollView` 内部嵌套可滚动子控件。

## 2. 判定原则

- 谁更接近用户意图，谁消费手势。
- 以 `ACTION_DOWN` 建立事件归属，再基于 `MOVE` 方向做切换判断。
- 一旦切换消费方，要正确处理 `ACTION_CANCEL`。

## 3. 外部拦截法（父容器主导）

思路：
- 父容器在 `onInterceptTouchEvent()` 中根据位移方向和阈值决定是否拦截。

优点：
- 规则集中，便于统一管理。

缺点：
- 父容器要理解子控件行为，维护成本高。

## 4. 内部拦截法（子控件主导）

思路：
- 子控件在合适时机调用 `requestDisallowInterceptTouchEvent(true)`。

优点：
- 子控件更了解自身滚动能力。

缺点：
- 多层嵌套时容易出现边界条件错乱。

## 5. 实战示例：外层横滑 + 内层横滑

策略：
- `DOWN` 先交给子控件尝试。
- `MOVE` 时若横向位移明显且子控件已到边界，再放权给父容器。

```kotlin
override fun onInterceptTouchEvent(ev: MotionEvent): Boolean {
    when (ev.actionMasked) {
        MotionEvent.ACTION_DOWN -> {
            lastX = ev.x
            lastY = ev.y
            parent.requestDisallowInterceptTouchEvent(true)
        }
        MotionEvent.ACTION_MOVE -> {
            val dx = ev.x - lastX
            val dy = ev.y - lastY
            if (kotlin.math.abs(dx) > kotlin.math.abs(dy) && shouldParentConsume(dx)) {
                parent.requestDisallowInterceptTouchEvent(false)
            }
        }
    }
    return super.onInterceptTouchEvent(ev)
}
```

## 6. 调试方法

- 打印 `dispatchTouchEvent` / `onInterceptTouchEvent` / `onTouchEvent` 的 action 和返回值。
- 重点看 `DOWN` 是否被消费、`MOVE` 何时被父容器截断、是否收到 `CANCEL`。
- 对比不同机型的触摸阈值和惯性行为。

## 7. 面试高频问题

### Q1: 为什么 `requestDisallowInterceptTouchEvent(true)` 对 `DOWN` 不生效？
A: 因为 `DOWN` 是事件序列起点，父容器在这一帧还未进入后续拦截决策流程。

### Q2: 滑动冲突解决的本质是什么？
A: 本质是为同一组事件建立稳定归属规则，并在边界处可控地切换归属。

### Q3: 为什么会出现“偶现无法滑动”？
A: 常见是阈值判断不稳定、边界判断错误或 `CANCEL` 状态处理不完整。

### Q4: 外部拦截和内部拦截怎么选？
A: 规则由父容器统一控制选外部拦截；子控件行为更复杂、更清楚边界时选内部拦截。

### Q5: 只看 `dispatchTouchEvent` 就能定位冲突吗？
A: 不够。必须同时看 `onInterceptTouchEvent` 和 `onTouchEvent` 的返回值链路。

## 8. 小结

实战里不要只背调用链，要先定义“谁在什么条件下接管手势”，再把阈值、边界、`CANCEL` 三件事处理完整。
