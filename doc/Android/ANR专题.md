# Android ANR专题

ANR 的本质是关键线程在限定时间内没有响应。面试要讲清楚触发条件、排查方法和治理优先级。

## 1. 什么是 ANR？

- ANR 全称 Application Not Responding。
- 常见原因是主线程长时间阻塞，导致输入、广播或服务处理超时。

## 2. 常见触发场景

- 主线程执行网络/磁盘 IO。
- 冷启动阶段初始化过重。
- 主线程等待锁或等待子线程结果。
- BroadcastReceiver / Service 处理逻辑过长。
- Binder 调用阻塞导致主线程卡死。

## 3. 排查路径

1. 获取 `traces.txt`，优先看 main 线程栈。
2. 判断是 CPU 忙、锁等待、IO 阻塞还是 Binder 阻塞。
3. 对照线程池、数据库、网络调用日志定位上游根因。
4. 用 Perfetto/Profiler 验证修复是否有效。

## 4. 主线程栈怎么读

- `Blocked`：大概率在等锁。
- `Waiting`：可能在等条件或结果。
- `Runnable` 但长时间不返回：可能在做重计算或循环。

## 5. 治理策略

- 严禁主线程 IO 与重计算。
- 生命周期回调只保留轻逻辑。
- 避免在主线程 `Future.get()`/`CountDownLatch.await()`。
- 锁粒度变小，避免全局大锁。
- 高风险路径加超时和降级。

## 6. 线上治理建议

- 建立 ANR 类型分类看板（输入、广播、服务、内容提供者等）。
- 按场景优先级治理：高频路径优先。
- 修复后灰度验证，观察 ANR 率与页面耗时联动变化。

## 7. 面试高频问题

### Q1: ANR 一定是主线程问题吗？
A: 触发点通常是主线程无响应，但根因可能在锁竞争、Binder 对端阻塞或后台线程卡死后主线程等待。

### Q2: `traces.txt` 最先看什么？
A: 先看 main 线程当前调用栈和状态，再看与之相关的锁持有线程。

### Q3: 为什么冷启动容易出 ANR？
A: 因为初始化集中在主线程，任何同步 IO、反射或锁等待都可能放大阻塞。

### Q4: StrictMode 能解决 ANR 吗？
A: 不能直接解决，但能提前暴露主线程 IO 等高风险点。

### Q5: 如何区分“卡顿”和“ANR”？
A: 卡顿是帧级别掉帧，ANR 是系统级无响应超时；卡顿严重后可能升级为 ANR。

## 8. 小结

ANR 处理的关键是“主线程栈优先、锁链路第二、场景复现第三”。定位准确比盲目异步化更重要。
