# Android 网络安全与HTTPS

网络题不仅问 Retrofit/OkHttp，也会问 HTTPS、证书校验、抓包风险和接口防重放。

## 1. HTTPS 基础

- HTTP + TLS 加密通道。
- 解决传输中窃听和篡改风险。
- 仍需服务端与客户端共同保证业务安全。

## 2. 常见风险

- 中间人攻击（MITM）
- 弱证书校验
- 明文日志泄漏敏感信息
- Token 重放与接口刷调用

## 3. Android 侧实践

- 全量接口走 HTTPS。
- 避免信任所有证书的临时代码进入生产。
- 对高敏感接口可做证书固定（Pinning）。
- 网络日志按环境分级，生产环境脱敏。

## 4. 证书固定要点

- 固定公钥或证书指纹。
- 支持证书轮换策略。
- 失败时要有可控降级方案，避免全量不可用。

## 5. 常见面试题

### Q1: HTTPS 一定绝对安全吗？
A: 不是。它保障传输链路安全，但业务层仍可能有鉴权、重放、越权等风险。

### Q2: 为什么“信任全部证书”是高危操作？
A: 会直接绕过证书校验，等于主动开放中间人攻击入口。

### Q3: 证书固定会带来什么代价？
A: 安全提升但运维复杂度上升，需要证书轮换和异常回退机制。

### Q4: 抓包工具能否抓到 HTTPS？
A: 在受控环境下可能通过安装代理证书实现，所以客户端仍要做严格校验和敏感数据保护。

### Q5: 网络安全治理和性能冲突吗？
A: 有权衡，但通常可通过连接复用、缓存和合理配置把性能损耗控制在可接受范围。

## 6. 小结

网络安全题要体现工程权衡：既要安全底线，也要可运维和可灰度，不做“绝对安全”的空泛回答。
