# Android 启动流程与冷启动

启动优化是 Android 高频面试题，常见问法包括：应用从点击图标到页面显示经历了什么、冷启动为什么慢、怎么做启动优化、如何量化首屏时间。

## 1. 启动类型

### 冷启动

- 应用进程不存在，需要从零创建进程、初始化 Application、启动首个 Activity。
- 开销最大，是性能优化重点。

### 温启动

- 进程还在，但 Activity 可能需要重建。
- 比冷启动快，但仍可能涉及页面恢复和数据重建。

### 热启动

- 应用进程和 Activity 都还在，只是从后台回到前台。
- 速度最快。

## 2. 冷启动的大致流程

1. 用户点击桌面图标。
2. Launcher 通过 Binder 请求 AMS 启动目标 Activity。
3. AMS 判断应用进程是否存在。
4. 如果不存在，则通知 Zygote fork 新进程。
5. 新进程启动后执行 `ActivityThread.main()`，初始化主线程 Looper。
6. 创建 `Application`，执行 `attachBaseContext()` 和 `onCreate()`。
7. 创建首个 Activity，执行生命周期。
8. 首屏完成测量、布局、绘制后显示给用户。

## 3. 冷启动慢通常慢在哪里？

- `Application.onCreate()` 初始化过重。
- 首屏 Activity 的 `onCreate()` 做了大量同步任务。
- 首屏布局复杂、图片资源过大。
- 首屏依赖网络，阻塞渲染。
- 多 SDK 初始化、反射、类加载过多。

## 4. 启动优化的核心思路

### 4.1 主线程只做必须做的事

- Crash、路由、极少数核心初始化可以保留。
- 其他初始化尽量延后、异步化或懒加载。

### 4.2 首屏优先

- 优先让第一个可交互页面尽快显示。
- 数据可以分层加载，不必一次拉满。

### 4.3 任务编排

- 按依赖关系和优先级拆分启动任务。
- 区分主线程任务、IO 任务、延迟任务。

### 4.4 量化优化

- 没有埋点和基线的“优化”大概率只是主观感觉。
- 常见指标有启动耗时、首帧时间、完全显示时间。

## 5. 如何度量启动时间？

常见口径：

- 点击图标到首帧绘制完成。
- 点击图标到首屏可交互。
- 点击图标到关键数据加载完成。

常用工具：

- `adb shell am start -W`
- Android Studio Profiler
- Perfetto / Systrace
- 自研埋点

## 6. SplashScreen 要怎么理解？

- Android 12+ 有系统级 SplashScreen。
- 它解决的是视觉过渡一致性，不等于真正意义上的启动优化。
- 如果后台初始化过重，SplashScreen 只是在“遮住慢”，不是“消除慢”。

## 7. 常见优化手段

### 延迟初始化

- 非核心 SDK 放到首帧之后。
- 可结合 IdleHandler、协程、线程池做延后执行。

### 懒加载

- 用户没进入某个模块前，不要提前初始化整套能力。

### 预加载

- 对确定高频使用的数据或资源做适度预取。
- 但预加载要克制，否则会转化成启动负担。

### 布局优化

- 减少层级。
- 避免首屏大图和复杂自定义 View。

## 8. 面试高频问题

### Q1: 应用启动时 `Application` 和首个 Activity 谁先执行？
A: 先创建 Application，再创建首个 Activity。Application 是进程级初始化入口。

### Q2: 冷启动优化最有效的手段是什么？
A: 通常是精简主线程初始化和首屏解耦。因为绝大多数冷启动慢都不是某个神秘系统问题，而是应用自己在启动期做了太多事。

### Q3: 为什么说不要把所有 SDK 都放在 `Application.onCreate()`？
A: 因为这会把所有初始化成本都压到启动最敏感的时段，直接拉长冷启动时间。

### Q4: `adb shell am start -W` 能看什么？
A: 能看到 `ThisTime`、`TotalTime`、`WaitTime` 等启动相关指标，适合快速对比不同版本的启动耗时。

### Q5: 启动优化和骨架屏、闪屏页是一回事吗？
A: 不是。骨架屏和闪屏更偏体验层兜底，而启动优化的本质是减少真实耗时和主线程阻塞。

## 9. 小结

冷启动题的回答框架最好固定为：启动链路、耗时热点、优化手段、量化方式。这样既有原理，又有工程落地感。
