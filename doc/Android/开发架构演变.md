# 开发架构



## MVC（Model - View - Controller）



**Model** 包括了所有**数据模型与数据状态**，同时负责数据的交互与存储；

**View** 包括了**呈现给用户的UI以及其他所有和用户的交互**，如传统的XML布局

**Controller** 包括了**业务逻辑以及Model与View之间的交互**，它像是Model与View之间的桥梁，通过Controller可以去操作Model并返回结果显示在View上,在开发中,Activity承载了Controller的角色，这会让Activiity非常的臃肿,容易变成数千行的"死亡类"



### MVC 的优缺点总结

| **维度**   | **特点**                                                     |
| ---------- | ------------------------------------------------------------ |
| **优点**   | 逻辑清晰，容易理解，适合小型项目。                           |
| **缺点**   | **Controller 过重**：在 Android 中，Activity 既要处理 View（生命周期、UI 寻找）又要处理 Controller 逻辑，容易变成数千行的“死亡类”。 |
| **耦合度** | View 和 Controller 耦合严重，单元测试较难开展。              |



### 代码示例：简单计数器

这个例子展示了一个点击按钮增加数字的简单逻辑。

#### 1. Model (模型层)

模型层不应该包含任何 Android 框架的代码。

Java

```java
// CounterModel.java
public class CounterModel {
    private int count = 0;

    public int getCount() {
        return count;
    }

    public void increment() {
        count++;
    }
}
```

#### 2. View (视图层)

视图通常就是你的 XML 文件。

XML

```java
<LinearLayout ... orientation="vertical">
    <TextView
        android:id="@+id/countText"
        android:text="0"
        android:textSize="30sp" />

    <Button
        android:id="@+id/incrementButton"
        android:text="增加" />
</LinearLayout>
```

#### 3. Controller (控制层)

`Activity` 充当 Controller，它持有 Model 并操作 View。

Java

```java
// MainActivity.java
public class MainActivity extends AppCompatActivity {
    private CounterModel model; // 持有 Model
    private TextView countText;
    private Button incrementButton;

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_main);

        // 初始化 Model
        model = new CounterModel();

        // 绑定 View
        countText = findViewById(R.id.countText);
        incrementButton = findViewById(R.id.incrementButton);

        // 处理用户交互
        incrementButton.setOnClickListener(v -> {
            // 1. 调用 Model 处理业务逻辑
            model.increment();
            
            // 2. 更新 View 显示结果
            updateUI();
        });
    }

    private void updateUI() {
        countText.setText(String.valueOf(model.getCount()));
    }
}
```



## MVP（Model-View-Presenter）

在 MVP 中，Presenter 和 View 通过 **接口 (Interface)** 进行通信，这使得代码非常易于进行单元测试。

### 代码示例：登录功能实践

#### 1. 定义契约 (Contract)

这是一个最佳实践，将 View 和 Presenter 的接口写在一起，一眼就能看清业务逻辑。



```java
public interface LoginContract {
    interface View {
        void showLoginSuccess(String message);
        void showLoginError(String error);
    }

    interface Presenter {
        void login(String username, String password);
    }
}
```

#### 2. Model (模型层)

负责数据处理，依然不涉及任何 Android 系统 API。



```java
public class UserManager {
    public boolean authenticate(String user, String pass) {
        return "admin".equals(user) && "123".equals(pass);
    }
}
```

#### 3. Presenter (中间层)

它持有 View 的接口引用和 Model 的引用。**它是纯 Java/Kotlin 代码，不包含 `Context`。**



```java
public class LoginPresenter implements LoginContract.Presenter {
    private LoginContract.View view; // View 接口
    private UserManager model;       // Model

    public LoginPresenter(LoginContract.View view) {
        this.view = view;
        this.model = new UserManager();
    }

    @Override
    public void login(String username, String password) {
        if (model.authenticate(username, password)) {
            view.showLoginSuccess("欢迎回来！");
        } else {
            view.showLoginError("用户名或密码错误");
        }
    }
}
```

#### 4. View (视图层 - Activity)

Activity 只需要负责 UI 的初始化和 Presenter 的调用，并实现 View 接口。

Java

```java
public class LoginActivity extends AppCompatActivity implements LoginContract.View {
    private LoginPresenter presenter;
    private EditText userEdit, passEdit;

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_login);

        presenter = new LoginPresenter(this); // 初始化 Presenter
        
        findViewById(R.id.loginBtn).setOnClickListener(v -> {
            presenter.login(userEdit.getText().toString(), passEdit.getText().toString());
        });
    }

    @Override
    public void showLoginSuccess(String message) {
        Toast.makeText(this, message, Toast.LENGTH_SHORT).show();
    }

    @Override
    public void showLoginError(String error) {
        userEdit.setError(error);
    }
}
```



###  优点

- **彻底解耦**：View（Activity/Fragment）不再直接操作 Model。所有的业务逻辑都封装在 Presenter 中，View 只负责 UI 的展示和用户事件的传递。

- **极高的可测试性**：这是 MVP 最大的杀手锏。由于 Presenter 通过接口与 View 交互，且不持有 Android 框架代码（如 Context），你可以直接编写 **JUnit 单元测试**，而无需启动模拟器。

- **代码复用性**：一个 Presenter 逻辑可以驱动多个实现相同接口的 View。例如，手机版和平板版的 UI 不同，但可以共享同一个登录逻辑。

- **职责清晰**：

  - **Model**：拿数据。

  - **View**：画界面。

  - **Presenter**：写逻辑。

### 缺点

**接口爆炸（Interface Overhead）**： 为了实现解耦，每个页面至少需要定义一个 `Contract` 类，包含 `View` 和 `Presenter` 两个接口。对于简单的页面，这会显著增加类文件的数量和开发工作量。

**生命周期管理风险（内存泄漏）**： Presenter 持有 View 的引用。如果用户在后台异步请求数据时关闭了 Activity，而 Presenter 仍在运行，就会导致 **Activity 无法被回收（内存泄漏）**。此外，当回调发生时调用 `view.showData()` 会触发 NullPointerException。

**双向耦合**： 虽然解耦了 UI 实现，但 View 和 Presenter 之间存在强依赖关系（通常是 1:1）。当 View 需要增加一个功能时，往往要修改接口、Presenter 和 Activity 三处代码。

**代码冗余**： 相比于现代的 MVVM（数据驱动），MVP 需要大量的手动代码来同步状态（比如不停地调用 `view.showLoading()`、`view.hideLoading()`）。



针对 MVP 的原生缺陷（尤其是**内存泄漏**和**接口冗余**），可以通过引入 **Base 类封装**、**生命周期感知**以及**弱引用**来显著提升架构的健壮性。

以下是主流的改进方案：

------

### 1. 解决内存泄漏：生命周期绑定

最常见的问题是异步回调时 Activity 已销毁。我们可以通过 `BasePresenter` 统一管理 View 的生命周期。

Java

```java
// 基础 Presenter 接口
public interface IBasePresenter<V> {
    void attachView(V view);   // 绑定 View
    void detachView();         // 解绑 View
    boolean isViewAttached();  // 判断 View 是否还在
}

// 抽象实现类
public abstract class BasePresenter<V> implements IBasePresenter<V> {
    protected Reference<V> mViewRef; // 使用弱引用防止内存泄漏

    @Override
    public void attachView(V view) {
        mViewRef = new WeakReference<>(view);
    }

    protected V getView() {
        return mViewRef != null ? mViewRef.get() : null;
    }

    @Override
    public boolean isViewAttached() {
        return mViewRef != null && mViewRef.get() != null;
    }

    @Override
    public void detachView() {
        if (mViewRef != null) {
            mViewRef.clear();
            mViewRef = null;
        }
    }
}
```

------

### 2. 自动化管理：利用 Jetpack Lifecycle

手动在每个 Activity 的 `onDestroy` 中调用 `detachView` 很麻烦且容易忘。可以利用 Android 官方的 `LifecycleObserver` 让 Presenter 自动感知生命周期。

Java

```java
public class LoginPresenter extends BasePresenter<LoginContract.View> implements LifecycleObserver {

    @OnLifecycleEvent(Lifecycle.Event.ON_DESTROY)
    public void onDestroy() {
        detachView(); // 自动解绑，无需在 Activity 显式调用
    }
}

// 在 Activity 中只需一行
getLifecycle().addObserver(presenter);
```

------

### 3. 减少接口爆炸：泛型与契约类

不要为每个简单的页面都写一套逻辑，提取通用的 `BaseView`。

Java

```java
public interface BaseView {
    void showLoading();
    void hideLoading();
    void onError(String msg);
}

// 契约类依然保留，但继承通用接口
public interface LoginContract {
    interface View extends BaseView {
        void onLoginSuccess();
    }
    // ...
}
```



## MVVM（Model-View-ViewModel）

通过 **可观察数据 + 生命周期感知** 实现自动同步。



ViewModel不持有View的引用，而是暴露出一个（数据源）通过 **Data Binding（数据绑定）** 或 **LiveData/Flow** 实现了视图与数据的**自动同步**。

 **Data Binding（数据绑定）** 需要再XML写表达式，个人不喜欢这种方式

### 1. MVVM 最小实践代码

我们将之前“计数器”的例子用 MVVM 实现。这里使用官方的 `ViewModel` 和 `LiveData` 组件。

#### Model (模型层)

Java

```java
// CounterModel.java
public class CounterModel {
    public int getNewCount(int currentCount) {
        return currentCount + 1;
    }
}
```

#### ViewModel (视图模型层)

**核心点：** ViewModel 绝对不包含任何 `android.widget.*` 的引用，它只负责维护 `LiveData` 状态。

Java

```java
// CounterViewModel.java
import androidx.lifecycle.MutableLiveData;
import androidx.lifecycle.ViewModel;

public class CounterViewModel extends ViewModel {
    private final CounterModel model = new CounterModel();
    
    // 使用 LiveData 包装数据
    private final MutableLiveData<Integer> countData = new MutableLiveData<>(0);

    public MutableLiveData<Integer> getCountData() {
        return countData;
    }

    public void increment() {
        Integer current = countData.getValue();
        if (current != null) {
            countData.setValue(model.getNewCount(current));
        }
    }
}
```

#### View (视图层)

Activity 只需要告诉 ViewModel “我想观察数据”和“我点了按钮”。

Java

```java
// MainActivity.java
public class MainActivity extends AppCompatActivity {
    private CounterViewModel viewModel;

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_main);

        TextView countText = findViewById(R.id.countText);
        Button btn = findViewById(R.id.incrementButton);

        // 1. 获取 ViewModel 实例（生命周期由系统管理）
        viewModel = new ViewModelProvider(this).get(CounterViewModel.class);

        // 2. 观察数据变化 (观察者模式)
        viewModel.getCountData().observe(this, count -> {
            countText.setText(String.valueOf(count));
        });

        // 3. 处理交互
        btn.setOnClickListener(v -> viewModel.increment());
    }
}
```

以上代码，还是需要手动执行setText方法

如果使用DataBinding,在XML布局中配置规则，则无需手动set相应方法

### MVVM 进阶实践：DataBinding + ViewModel

#### 1. 开启 DataBinding (build.gradle)

Gradle

```java
android {
    buildFeatures {
        dataBinding true
    }
}
```

#### 2. XML 布局 (activity_main.xml)

使用 `<layout>` 标签包裹，直接将 ViewModel 绑定进布局。

XML

```
<layout xmlns:android="http://schemas.android.com/apk/res/android">
    <data>
        <variable
            name="vm"
            type="com.example.CounterViewModel" />
    </data>

    <LinearLayout ...>
        <TextView
            android:text="@{String.valueOf(vm.countData)}" 
            ... />

        <Button
            android:onClick="@{() -> vm.increment()}"
            android:text="增加" />
    </LinearLayout>
</layout>
```

#### 3. Activity (极致精简)

Activity 此时只负责“摆放”布局，不再干涉业务逻辑。

Java

```java
public class MainActivity extends AppCompatActivity {
    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        // 自动生成的 Binding 类
        ActivityMainBinding binding = DataBindingUtil.setContentView(this, R.layout.activity_main);
        CounterViewModel viewModel = new ViewModelProvider(this).get(CounterViewModel.class);
        
        binding.setVm(viewModel);
        binding.setLifecycleOwner(this); // 关键：让 XML 感知 LiveData 变化
    }
}
```



## MVI（Model - View - Intent）

`单向数据流`

**Intent (意图)**：用户的所有操作（点击、滑动）都定义为 Intent。

**State (状态)**：UI 的唯一真实数据源。一个页面只有一个 State 对象。

**Reducer**：接收旧状态和 Intent，计算出新状态。



### MVI 进阶实践：使用 Channel 传递 Intent

#### 1. 定义契约 (不变)

Kotlin

```kotlin
data class MainState(val count: Int = 0, val isLoading: Boolean = false)
sealed class MainIntent {
    object Increment : MainIntent()
    object Decrement : MainIntent()
}
```

#### 2. ViewModel：使用 Actor 模式处理 Channel

我们通过 `Channel` 接收意图，并在 `init` 块中持续监听（Consume）它。

Kotlin

```kotlin
class MainViewModel : ViewModel() {

    // 1. 创建 Intent 的 Channel
    private val intentChannel = Channel<MainIntent>(Channel.UNLIMITED)

    // 2. 暴露给 View 的唯一状态源
    private val _state = MutableStateFlow(MainState())
    val state: StateFlow<MainState> = _state

    init {
        // 3. 在协程中持续监听 Intent
        handleIntents()
    }

    // View 调用此方法发送意图
    fun sendIntent(intent: MainIntent) {
        viewModelScope.launch {
            intentChannel.send(intent)
        }
    }

    private fun handleIntents() {
        viewModelScope.launch {
            // 4. 将 Channel 转化为 Flow 进行处理
            intentChannel.consumeAsFlow().collect { intent ->
                when (intent) {
                    is MainIntent.Increment -> reduce { it.copy(count = it.count + 1) }
                    is MainIntent.Decrement -> reduce { it.copy(count = it.count - 1) }
                }
            }
        }
    }

    // 辅助方法：确保状态转换的原子性
    private fun reduce(reducer: (MainState) -> MainState) {
        _state.value = reducer(_state.value)
    }
}
```



### 3. View 层 (Activity) 发送意图

在 View 层，我们不再是调用一个函数，而是向一个管道“塞入”动作。

Kotlin

```kotlin
class MainActivity : AppCompatActivity() {
    private val viewModel: MainViewModel by viewModels()

    override fun onCreate(savedInstanceState) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_main)

        // 发送 Intent：就像往信箱里投信
        btnPlus.setOnClickListener {
            viewModel.sendIntent(MainIntent.Increment)
        }

        // 接收 State：就像看信箱里的回复
        lifecycleScope.launch {
            viewModel.state.collect { state ->
                tvCount.text = state.count.toString()
            }
        }
    }
}
```
