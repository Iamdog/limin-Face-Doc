# Android 崩溃治理专题

稳定性面试里，崩溃治理是核心模块。重点是“监控、归因、止损、预防”闭环。

## 1. 崩溃分类

- Java Crash（NPE、数组越界等）
- Native Crash（SIGSEGV 等）
- ANR（无响应）

## 2. 治理闭环

1. 采集：统一崩溃平台，保证堆栈和设备上下文完整。
2. 归因：按版本、机型、页面、业务链路聚类。
3. 止损：热修、降级、开关、灰度回滚。
4. 预防：测试策略、静态检查、核心路径守护。

## 3. 工程实践

- 建立 Top 崩溃看板，按影响用户数排序。
- Crash Free Users 作为核心指标之一。
- 高风险模块发布前强制专项回归。

## 4. 常见面试题

### Q1: 如何定义“治理有效”？
A: 看影响用户规模和趋势是否持续下降，而不是只看单条崩溃修复数量。

### Q2: 为什么修了很多崩溃但指标没明显改善？
A: 可能修复的是低频长尾，未命中高影响 Top 崩溃。

### Q3: Native Crash 怎么排查？
A: 结合 tombstone、符号表和版本映射做地址还原，再定位触发路径。

### Q4: 如何避免新版本稳定性回退？
A: 分阶段灰度 + 关键指标门禁 + 可快速回滚机制。

### Q5: 崩溃和 ANR 可以分开治理吗？
A: 可以分模块治理，但稳定性体系应统一看板和优先级评估。

## 5. 小结

崩溃治理是长期工程，关键是用数据驱动优先级，而不是“见一个修一个”。
